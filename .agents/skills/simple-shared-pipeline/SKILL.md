# simple-shared-pipeline

## Purpose
Keep future pipeline handling lightweight and suitable for a small boutique.

## Use when
Later app pipeline/status decisions.

## Rules
- One shared pipeline is acceptable because volume is low.
- Keep statuses minimal and readable.
- Prefer dates/history over advanced SLA mechanics.

## Recommended statuses
- new
- after contact
- follow-up
- in progress
- completed

## Guardrails
- No huge status matrix.
- No enterprise pipeline logic.

## Output format
- Status set
- Why it stays simple
- Where special cases should live instead
